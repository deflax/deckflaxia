# Fix JUCE Smoke Screenshot Rendering

## TL;DR
> **Summary**: Change `--juce-ui-smoke-test --screenshot` from a synthetic headless panel painter to a deterministic 1920×1080 real `MainComponent` render so generated evidence matches the actual Deckflaxia GUI content. Add a smoke-log marker and strengthen CTest verification so this path cannot silently regress.
> **Deliverables**:
> - CLI screenshot path renders real `MainComponent` via `paintEntireComponent(...)`
> - `JuceUi.Screenshot` evidence uses deterministic 1920×1080 output
> - Screenshot log includes `screenshot-source: live-main-component` and `screenshot-size: 1920x1080`
> - Targeted CTest/build verification passing or explicitly blocked with evidence
> **Effort**: Short
> **Parallel**: NO
> **Critical Path**: Task 1 → Task 2 → Task 3

## Context
### Original Request
The user observed that `.omo/evidence/real-playable-juce/task-gui-layout-after.png` generated by `--screenshot` differs from the actual running GUI screenshot `.omo/evidence/real-playable-juce/Screenshot_20260528_100038.png`, then asked to fix the `--screenshot` path.

### Interview Summary
- Actual live runtime screenshot is visually correct: deck grid, utility column, sidebar, and status bar are distinct and non-overlapping.
- Generated `--screenshot` evidence appears visually different because it is produced by a synthetic painter.
- User chose deterministic `1920x1080` for CTest screenshot evidence.
- User chose to add a smoke-log proof marker.

### Metis Review (gaps addressed)
- Locked canonical screenshot region to `MainComponent` content, not native OS title bar/window chrome.
- Added deterministic `1920x1080` size and explicit marker assertions.
- Added guardrails to avoid UI redesign, display-dependent screenshots, and broad visual-regression framework scope.
- Added failure/unavailable path preservation and executable acceptance criteria.

## Work Objectives
### Core Objective
Make `Deckflaxia --juce-ui-smoke-test --screenshot <path> --exit-after-init` render the real `deckflaxia::ui::MainComponent` component tree at deterministic `1920x1080`, instead of using the synthetic `paintHeadlessSmokeScreenshot(...)` path, and prove that behavior in logs/tests.

### Deliverables
- `src/app/Main.cpp`: route JUCE UI smoke screenshot requests through real component rendering.
- `src/ui/JuceComponentTree.cpp` and `src/ui/JuceComponentTree.h`: expose/adjust a helper that captures `MainComponent` at `1920x1080` and emits marker/dimension output.
- `CMakeLists.txt` and/or `tests/ui/JuceUiSmokeTests.cpp`: strengthen `JuceUi.Screenshot` verification for marker, PNG creation, and `1920x1080` dimensions.
- Evidence under `.omo/evidence/real-playable-juce/` showing command output and generated PNG.


- `.omo/evidence/real-playable-juce/task-fix-juce-screenshot.png` is a valid PNG with dimensions `1920 x 1080`.

### Must Have
- Screenshot output must render real `MainComponent`, not the synthetic `paintHeadlessSmokeScreenshot(...)` panel collage.
- CTest screenshot size must be deterministic `1920x1080`.
- The smoke command must emit a machine-checkable marker: `screenshot-source: live-main-component`.
- The smoke command must emit a machine-checkable size line: `screenshot-size: 1920x1080`.
- `--juce-ui-smoke-test --dump-components --exit-after-init` must continue to output the component tree as before.
- Non-JUCE/unavailable screenshot behavior must remain honest and non-successful for requested screenshots.

### Must NOT Have
- No UI layout redesign.
- No new DJ controls, audio behavior, browser behavior, plugin behavior, persistence behavior, or playlist behavior.
- No display-dependent screenshot dimensions for CTest.
- No reliance on manual visual confirmation as the primary pass/fail criterion.
- No broad golden-image or visual regression framework unless explicitly requested later.
- No deleting the synthetic helper unless all call sites are proven obsolete.
